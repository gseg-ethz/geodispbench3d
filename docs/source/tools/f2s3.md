# F2S3

F2S3 is a feature-to-supervoxel cross-correlation tool driven via its CLI.
Bench wiring lives in
[`src/geodispbench3d_f2s3/`](https://github.com/gseg-ethz/geodispbench3d/tree/main/src/geodispbench3d_f2s3/). It is the
**canonical `CliToolAdapter` example**: an external binary run one process per
trial via `conda run`, with subprocess isolation so a tool crash fails a single
trial rather than the whole sweep.

## How to obtain F2S3

F2S3 is not bundled with geodispbench3d. It lives in its own repository — build
and install it from
[gseg-ethz/F2S3_pc_deformation_monitoring](https://github.com/gseg-ethz/F2S3_pc_deformation_monitoring),
following that repo's README for build and environment setup. The benchmark
expects F2S3 to be invokable inside a dedicated conda env named `f2s3-dev312`
(see [How F2S3 plugs in](#how-f2s3-plugs-in) below).

If the `f2s3-dev312` env — or conda itself — cannot be resolved before the first
trial, the `CliToolAdapter` preflight raises a `ToolPreflightError` that appends
the tool YAML's `remediation` hint and `help_url`. Both are configured in
[`f2s3.yaml`](https://github.com/gseg-ethz/geodispbench3d/blob/main/src/geodispbench3d_f2s3/conf/tool/f2s3.yaml); the `help_url`
points at the gseg-ethz repo above. Note the deliberate limitation (D-02): the
preflight validates conda and the named env, but does **not** spawn the env to
confirm the trailing `f2s3` binary exists inside it. A binary missing *inside*
the env is surfaced by the first trial's nonzero-exit handling
(`error_kind="nonzero_exit"` / `"entry_not_found"`), not by the preflight.

## Run the default suite

```bash
conda run -n iof3d_cosicorr3d-dev312 \
  geodispbench3d run benchmarks/suites/f2s3_voxel_refine.yaml
```

This sweeps `voxel_grid_size ∈ {0.0, 0.2, …, 2.0}` × `refine_results ∈ {false, true}`
(22 combinations) on the Mattertal scans, minimizing median 3D displacement
error.

## How F2S3 plugs in

F2S3 lives in its own conda env (`f2s3-dev312`), so the tool YAML invokes it
via `conda run`:

```yaml
entry: conda run -n f2s3-dev312 f2s3
```

`conda run -n f2s3-dev312 f2s3` is the canonical default entry. Running the
`f2s3` binary directly (already on `PATH`, no `conda run` wrapper) is a
supported alternative — set `entry: f2s3` if F2S3 is installed into the same
env geodispbench3d runs in.

Boolean parameters that F2S3 expects as argparse `store_true` flags
(`--save_interim`, `--refine_results`) are listed in
`invocation.presence_flag_params:` — the adapter renders them as `--<name>`
when the value is truthy and omits them when falsy. Static, always-on flags
(input scans, `--save_interim`) are declared in `invocation.static_params:`
so they don't pollute the swept parameter space.

## Per-trial timeout

A long-running F2S3 trial can be bounded with the optional
`execution.timeout_seconds` knob (D-04). It is **opt-in**: the shipped
[`f2s3.yaml`](https://github.com/gseg-ethz/geodispbench3d/blob/main/src/geodispbench3d_f2s3/conf/tool/f2s3.yaml) leaves it unset
(commented out) because F2S3 sweeps can legitimately run for hours, so there is
no shipped ceiling. Unset (or `<= 0`) means no timeout.

```yaml
execution:
  mode: subprocess
  in_process_safe: false
  timeout_seconds: 7200   # opt-in per-trial wall-clock cap (seconds)
```

A `--timeout` flag on `geodispbench3d run` overrides the YAML value for that
invocation (it writes through the adapter's timeout seam).

**Termination guarantee.** On expiry the adapter kills the **whole tool process
tree**, including the `conda run` descendants, via process-group `SIGKILL`
(`os.killpg`) on POSIX. On non-POSIX platforms it falls back to terminating the
direct child process only. A `ProcessLookupError` during the kill (the process
already exited) is swallowed; the trial is still recorded as a timeout. The docs
make no stronger cleanup claim than this: process-group termination on POSIX,
direct-child fallback elsewhere. An individual timeout is **non-fatal** to the
sweep — the run continues to the next trial and the timeout is reported on its
own line (see [the subprocess contract](../integrating/cli-tool.md#subprocess-contract)).

## Hashed run dirs

Each trial gets its own subfolder under the tool YAML's `outputs.hashed_run_dir.root`:

```yaml
outputs:
  hashed_run_dir:
    root: /scratch/00_data/geodispbench3d/f2s3
    arg_name: --results_dir
    hash_length: 12
```

The adapter hashes the trial's parameters into a 12-char SHA256 prefix and
appends `--results_dir <root>/<hash>` to argv. Identical parameter sets
re-use the same directory, which is convenient for resuming a sweep.

## Output parsing

F2S3 writes one ASCII text file per tile under
`<run_dir>/output/` (or `<run_dir>/output/refined_results/` when
`--refine_results` is set). Each file has 7 whitespace-separated columns
in scientific notation, no header:

```
X1  Y1  Z1  X2  Y2  Z2  magnitude
```

[`parse_f2s3_output`](https://github.com/gseg-ethz/geodispbench3d/blob/main/src/geodispbench3d_f2s3/output_parser.py)
loads each tile via `pchandler.data_io.Csv.load` (specifying the column
names via the `scalar_fields=` kwarg), merges with `PointCloudData.merge`,
samples within 15 m of each GT epoch-1 position, and returns the
component-wise median of `(X2 - X1, Y2 - Y1, Z2 - Z1)` per GT label.

## Tuning the parser

The parser accepts:

```yaml
output_parser:
  fn: geodispbench3d_f2s3:parse_f2s3_output
  options:
    sample_radius_m: 15.0      # GT search radius
    aggregation: median        # median | mean
    prefer_refined: true       # use output/refined_results/ when present
```

If `prefer_refined: true` and `--refine_results` produced a non-empty
`refined_results/` directory, the parser uses that; otherwise it falls
back to plain `output/`. This means `refine_results: false` trials and
`refine_results: true` trials both read from the directory the tool
actually populated.

## Extending the parameter space

The CLI exposes more knobs than the default suite sweeps (e.g.
`--max_points_per_tile`, `--max_disp_magnitude`). Add them to the tool
YAML's `hyperparameters:` block; for choice parameters listed values flow
straight through; for range parameters, the adapter renders them as
`--<name> <value>`.

If a new parameter is a presence-only flag, add it to
`invocation.presence_flag_params:` so the adapter knows to render it as
`--<name>` (or omit it) rather than `--<name> true`.
